# MYXON Platform — Архитектура платформы

> Живой документ. Обновляется по мере принятия решений.
> Версия: 2026-04

---

## 1. Концепция

MYXON — промышленная IoT-платформа удалённого доступа к контроллерам
(климат, вентиляция, кормление и т.д.) в агросекторе.

**Ключевые принципы:**
- Клиент = единственный владелец своих данных
- Дилер не имеет доступа к данным клиента по умолчанию
- Деньги клиента → MYXON напрямую (сервис гарантирован независимо от дилера)
- Selling chain (финансы) ≠ Data access chain (доступ к данным)
- White-label поддержка для партнёров-производителей (HOTRACO и др.)

## 2. Иерархия участников

```
MYXON Platform  (platform_admin)
└── Partner  (напр. HOTRACO)           ← white-label, свой каталог устройств
    └── Dealer  (напр. Агровент)       ← региональный интегратор
        └── Customer  (ООО Бройлер)   ← конечный клиент
            ├── Site: Площадка А  (Краснодар)
            │   ├── Device: Orange Pi #1
            │   └── Device: Orange Pi #2
            ├── Site: Площадка Б  (Ростов)
            │   └── Device: Orange Pi #3
            └── Site: Площадка В  (Воронеж)
                └── Device: Orange Pi #4
```

### Тиры тенантов

| Tier | Кто | Создаёт | Видит |
|------|-----|---------|-------|
| `platform` | MYXON | партнёров, глобальные планы | всё |
| `partner` | HOTRACO и др. | дилеров, каталог устройств | своих дилеров + биллинг-метрики |
| `dealer` | Агровент и др. | клиентов, регистрирует Orange Pi | статус устройств своих клиентов (online/offline), биллинг |
| `customer` | Птицефабрика | площадки, пользователей | только свои данные |

---

## 3. Пользователи внутри Customer

Один customer-тенант может иметь несколько пользователей с разными правами:

| Роль | Доступ |
|------|--------|
| `customer_admin` | все площадки, управление пользователями, биллинг |
| `customer_engineer` | назначенные площадки, полный контроль устройств |
| `customer_viewer` | назначенные площадки, только просмотр |

Пользователь привязывается к одной или нескольким `Site` — это решает `customer_admin`.

## 4. Воркфлоу онбординга устройства

```
1. ПОДГОТОВКА  (Partner/Dealer)
   ├── Дилер прошивает Orange Pi (setup-debian.sh)
   ├── Серийник вносится в MYXON БД через dealer-портал
   └── Orange Pi отправляется клиенту (с серийником на наклейке)

2. ИНСТАЛЛЯЦИЯ  (Дилер / монтажник на месте)
   ├── Подключить контроллер к LAN-порту Orange Pi
   ├── Подключить Orange Pi к интернету (WAN)
   └── Включить питание → агент стартует автоматически (systemd)
       ├── Сканирует LAN → находит контроллер
       ├── Регистрируется на MYXON сервере
       └── Пробивает frpc-тоннель → устройство "online, unclaimed"

3. АКТИВАЦИЯ  (Клиент самостоятельно)
   ├── Клиент регистрируется в MYXON портале (или по инвайту)
   ├── ClaimWizard → вводит серийник с наклейки
   ├── Подтверждает → устройство привязано к его аккаунту
   ├── Указывает площадку (или создаёт новую)
   └── Подписка активируется → начинается биллинг

4. РАБОТА
   ├── Клиент видит все свои площадки и устройства
   ├── DeviceDashboard: HMI, аларм, лог событий
   └── Дилер видит только: онлайн/офлайн статус (для SLA)
```

## 5. Доступ дилера к данным клиента

**По умолчанию: нет доступа к операционным данным.**

```
Дилер видит:
  ✅ Устройство online/offline
  ✅ Количество устройств (для биллинга)
  ✅ Имя/контакт клиента (для поддержки)
  ❌ Аларм-детали
  ❌ HMI/экран контроллера
  ❌ Производственные данные
  ❌ Аудит-лог

Временный доступ поддержки:
  Клиент → "Разрешить дилеру просмотр на 24 часа"
  → Создаётся временный access grant с expiry
  → Клиент может отозвать досрочно
  → Все действия дилера логируются отдельно
```

## 6. Биллинг — денежный поток

### Принцип

```
Клиент платит MYXON напрямую (полная цена)
MYXON гарантирует сервис независимо от дилера
MYXON ежемесячно рассчитывается с дилером и партнёром
```

### Денежный поток

```
Клиент ──[2000 ₽/устройство/мес]──→ MYXON
                                        │
                          ежемесячный settlement
                                   ┌───┴───┐
                              Дилер         Partner
                           [1000 ₽]        [600 ₽]
                                        MYXON оставляет [400 ₽]
```

### Защита клиента

```
Дилер "забыл" заплатить MYXON → не важно, клиент уже заплатил MYXON
Дилер пропал → MYXON продолжает обслуживать клиента
               MYXON прекращает выплаты дилеру
               При необходимости переназначает поддержку
```

### Жизненный цикл неплатежа клиента

```
Клиент просрочил → MYXON уведомляет клиента
  → Grace period 7 дней
  → Режим read-only (данные видны, управление отключено)
  → Ещё 7 дней
  → Тоннель приостановлен (устройство offline в портале)
  → Дилер видит статус "suspended" (без деталей причины)
```

## 7. Модель данных биллинга

```
Plan
├── id, name  ("Basic", "Pro", "Enterprise")
├── price_per_device_per_month
└── revenue_shares[]
    ├── role: platform  → amount / percent
    ├── role: partner   → amount / percent
    └── role: dealer    → amount / percent

Subscription
├── device_id
├── plan_id
├── customer_tenant_id
├── selling_dealer_id       ← для начисления комиссии
├── selling_partner_id
├── status: active | suspended | cancelled
├── current_period_start
└── next_billing_date

LedgerEntry                 ← двойная запись
├── subscription_id
├── amount
├── currency
├── type: charge | platform_revenue | dealer_commission | partner_commission
├── tenant_id               ← кому относится запись
└── settled: bool

Settlement                  ← ежемесячная выплата
├── tenant_id               (дилер или партнёр)
├── period  (2024-12)
├── total_amount
├── entries: LedgerEntry[]
└── status: pending | paid | disputed
```

## 8. Техническая реализация биллинга

**Рекомендуется: Stripe Connect (Destination Charges)**

```
Customer → платёж через Stripe → деньги зачисляются на счёт MYXON
MYXON → Stripe Transfer API → автоматически переводит доли
Dealer → Stripe Connect аккаунт → видит свои выплаты в реальном времени
Partner → Stripe Connect аккаунт → аналогично
```

Это закрывает:
- PCI DSS compliance (карты не касаются наших серверов)
- SEPA, карты, SBP
- Автоматические налоговые документы
- Dispute management

## 11. Принятые решения (ADR)

| # | Решение | Причина |
|---|---------|---------|
| 1 | Клиент платит MYXON напрямую, не дилеру | Гарантия сервиса, защита клиента |
| 2 | Дилер не имеет доступа к данным клиента по умолчанию | Безопасность, конфиденциальность, GDPR |
| 3 | Selling chain ≠ Data access chain | Разделение финансов и доступа |
| 4 | Клиент активирует устройство самостоятельно | Клиент = владелец, без посредников |
| 5 | Orange Pi = агент с авто-дискавери контроллеров | Нет ручной конфигурации IP |
| 6 | frpc/frps тоннель — основной способ доступа | Устройства за NAT, без публичного IP |
| 7 | Stripe Connect для биллинга | PCI compliance, автоматические выплаты |
